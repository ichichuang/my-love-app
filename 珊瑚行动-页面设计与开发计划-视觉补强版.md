## 0. 结论摘要

本轮建议采用「单集合、多类型记录」方案：

- 继续使用固定数据库集合：`love_entries`
- 继续使用固定 `coupleId: "main"`
- 继续使用固定云存储前缀：`love-entries/main/`
- 新增 `kind` 字段区分：
  - `memory`：现有回忆记录
  - `song`：点歌清单
  - `task`：必做事项
- 旧回忆数据没有 `kind` 时，按 `memory` 兼容处理。
- 不新增 CloudBase collection。
- 不新增 cloud functions。
- 不新增 partner access。
- 不改变现有 CloudBase wrapper 架构。
- 页面只调用 repositories / composables，不直接调用 `wx.cloud`。

视觉上不建议大改设计系统基础层。现有 6 套策展配色、密度、字号、Wot UI 主题注入和运行时 CSS 变量能力已经足够支撑新功能。设计稿中的暖纸、红蓝、柔和状态色、卡片、标签、复选框等，都应映射为现有 `--app-*` token，而不是把设计稿中的 hex 或固定尺寸复制进页面。

本版在原计划基础上补充一条核心判断：

> 新页面不能只完成 CRUD 与清单能力，还必须呈现“私密情侣小程序”的页面情绪。点歌清单要像“她想听我唱的歌”，必做事项要像“我们想一起做的小事”，首页要像“私密记忆停靠处”，不能做成后台 dashboard、企业待办或普通歌单管理器。

因此，实现时需要同时满足两条线：

1. **架构线**：继续遵守 `page-meta + useNativeChromeSync + app-shell + repository + token` 的封版架构。
2. **情绪线**：通过温柔中文文案、红蓝小人线索、圆润纸张卡片、小便签 / 小标签 / 小印章感、轻微手绘装饰和完成后的亲密反馈，形成轻可爱、手绘感、情侣语境。

### 2.4 页面情绪与手绘视觉规范

#### 2.4.1 总体情绪定义

本项目的“可爱”不是儿童卡通、糖果色或商业插画，而是：

- 私密的可爱
- 温柔的可爱
- 手写感的可爱
- 两个人之间的小暗号
- 轻轻陪伴的可爱

页面整体应像一个长期可用的“私密记忆停靠处”。它可以有小标签、小便签、小印章、小勾选、小纸条，但不能变成儿童贴纸簿、营销海报或后台管理工具。

#### 2.4.2 三个新增功能的情绪目标

| 区域 | 情绪目标 | 不应变成 |
|---|---|---|
| 首页 | 今天想停靠在哪里；写回忆、点一首、勾一件都自然并列 | 后台 dashboard、数据看板、功能宫格工具页 |
| 点歌清单 | 她想听我唱的歌，唱过后轻轻收好 | 普通歌单管理器、音乐收藏夹、任务系统 |
| 必做事项 | 我们想一起做的小事，慢慢勾上 | 企业待办、项目管理、效率工具 |
| 回忆时间线 | 私密档案、普通日常也值得留下 | 公开 feed、社交动态、相册后台 |
| 空态 | 小内容还在路上，用温柔语气邀请开始 | 冷冰冰的“暂无数据” |
| 删除确认 | 克制、明确、保护性强 | 过度卖萌、吓人、系统后台提示 |

#### 2.4.3 允许的轻可爱表达

允许使用：

- 圆润卡片
- 柔和纸张底色
- 红蓝小人视觉线索
- 小标签、小便签、小印章感
- 温柔中文文案
- 轻微手绘线条感
- 完成后的亲密反馈
- 小圆点、小短线、小折角、小贴纸感
- 已完成事项轻轻淡化或轻柔划线
- 点歌状态像小纸条，而不是后台状态码

#### 2.4.4 禁止的可爱表达

禁止使用：

- 儿童卡通风
- 大量表情贴纸
- 高饱和糖果色
- 大面积商业 App 插画
- 海洋、珊瑚礁、贝壳、珍珠等元素
- 玻璃拟物
- 大量渐变
- 夸张动效
- 3D 质感
- 英文装饰文案
- 页面里直接写 raw SVG 颜色、hex、rgba、固定 rpx / px、直接阴影

#### 2.4.5 手绘风执行方式

手绘感不依赖大图堆叠，而通过轻量结构表达：

1. **红蓝线索**
   - 点歌偏 `--app-primary` / `--app-color-red-person`
   - 必做事项偏 `--app-accent` / `--app-color-blue-person`
   - 回忆记录保留红蓝平衡

2. **纸张感**
   - 卡片使用 `@include panel`
   - Hero 与强调区使用 `var(--app-surface-strong)`
   - 输入区使用 `var(--app-field)`
   - 不新增页面级 raw 背景色

3. **小便签 / 小印章**
   - 状态标签使用 `var(--app-radius-badge)` 或 `var(--app-radius-pill)`
   - 标签底色使用 `--app-primary-soft`、`--app-accent-soft`、`--app-success-soft`、`--app-warning-soft`
   - 文字色使用对应主 token

4. **手写式亲密感**
   - 文案更像两个人之间的小记录
   - 不用“管理”“配置”“提交任务”等后台语气
   - 不用英文缩写或技术词作为用户界面文案

5. **完成反馈**
   - 点歌已唱：`已经收进唱过的歌里。`
   - 事项勾选：`已经轻轻勾上。`
   - 取消完成：`已经放回未完成。`

#### 2.4.6 情侣语境边界

当前 Stage 1 仍是 owner-only，不能新增伴侣访问或账号绑定。但页面可以承载情侣语境：

允许：

- “她想听的歌”
- “我们想一起做的事”
- “已经一起完成”
- “唱给她听”
- “小约定”
- “小愿望”
- “慢慢完成”

禁止：

- “邀请她加入”
- “绑定伴侣账号”
- “发送给女朋友”
- “共享给伴侣”
- “双方可见”
- “评论”
- “公开分享”
- “发布动态”

文案必须表达“内容是关于两个人的”，但不能暗示当前已开放给两个人登录访问。

## 4. 设计变量与配色变量替换方案

### 4.1 总体原则

设计稿中的色彩是视觉说明，不能直接复制为页面样式。实现时必须使用现有 token：

- 不在 `src/pages/**` 写 hex。
- 不在 `src/components/**` 写 hex。
- 不在页面 / 组件里写 `rgba()`。
- 不在页面 / 组件里写固定 rpx / px。
- 不在页面 / 组件里写固定阴影。
- 不在页面 / 组件里写固定 transition duration。
- 不在页面 / 组件里定义新的 `--app-*`。

### 4.3 是否需要新增 token

当前结论：**第一版不新增 token。**

原因：

- 新增页面需要的间距、圆角、字号、状态色、阴影、按钮高度和卡片间距，现有 token 已覆盖。
- 复选框尺寸可用 `--app-control-height-sm` 或 Wot UI 内部尺寸。
- 筛选按钮可复用 `AppOptionGroup` / `AppOptionButton` 或现有按钮 token。
- 卡片可以复用 `@include panel`。
- 状态色已有 success / warning / danger / info。

如果实现中确实发现现有 token 不能支撑，例如需要长期复用的「清单复选框尺寸」或「功能入口卡片最小高度」，只能按以下流程新增：

1. 在 `src/design-system/token-registry.ts` 的 `componentVarNames` 中登记。
2. 在 `src/design-system/size-resolver.ts` 输出值。
3. 如果需要静态 fallback，同步 `src/styles/tokens/**`。
4. 更新 `docs/DESIGN_SYSTEM.zh-CN.md`。
5. 运行 `pnpm scan:design-tokens` 和 `pnpm type-check:strict`。

第一版建议不要走这条路。

实现原则：

### 4.6 轻可爱 / 手绘风的 token 化实现规则

#### 4.6.1 手绘装饰只能消费现有 token

任何手绘装饰、便签感、纸条感都必须用现有 token 实现，例如：

```scss
.card-mark {
  color: var(--app-primary);
  background: var(--app-primary-soft);
  border: var(--app-panel-border-width) solid var(--app-border);
  border-radius: var(--app-radius-badge);
  font: var(--app-font-caption);
}
```

不能写：

```scss
.card-mark {
  color: #A84E5A;
  border-radius: 23rpx;
  box-shadow: 0 8rpx 24rpx rgba(0, 0, 0, 0.12);
}
```

#### 4.6.2 功能情绪与 token 对应

| 情绪 | 推荐 token | 使用位置 |
|---|---|---|
| 轻轻停靠 | `--app-surface`、`--app-surface-strong` | 首页 hero、功能入口 |
| 小纸条 | `--app-field`、`--app-border`、`--app-radius-card` | 点歌卡片、事项卡片 |
| 她想听 | `--app-primary`、`--app-primary-soft` | 点歌状态、点歌入口 |
| 我们一起做 | `--app-accent`、`--app-accent-soft` | 事项状态、事项入口 |
| 已经完成 | `--app-success`、`--app-success-soft` | 已唱、已完成 |
| 暂缓一下 | `--app-warning`、`--app-warning-soft` | 暂缓曲目、待定日期 |
| 小心删除 | `--app-danger`、`--app-danger-soft` | 删除确认、危险区 |
| 私密说明 | `--app-text-soft`、`--app-text-muted` | 备注、空态说明 |

#### 4.6.3 小装饰策略

可以在组件内部加入 token 化结构装饰：

- `::before` 做小圆点
- `::after` 做轻折角
- 卡片头部做小标签
- 完成态做柔和淡化
- 功能入口中使用单个中文字 mark：`歌`、`约`、`忆`

注意：

- 装饰不能遮挡正文。
- 装饰不能依赖固定像素定位。
- 装饰不能造成深色模式对比度不足。
- 装饰不能新增 `--app-*` 局部变量。
- 如果使用组件局部变量，命名不要以 `--app-` 开头，除非加入 allowlist；第一版不建议使用。

#### 4.6.4 静态手绘素材规则

如果后续必须新增静态手绘素材：

- 放在 `static/` 下。
- 命名清晰，例如：
  - `static/doodle-song.png`
  - `static/doodle-task.png`
  - `static/doodle-memory.png`
- 不包含敏感信息。
- 不引用远程图片 URL。
- 不使用海洋、珊瑚礁、贝壳、珍珠元素。
- 不替代核心 UI 状态，图片加载失败时页面仍可用。
- 第一版优先使用已有 `static/logo-couple.png` 与 CSS 结构装饰，不急于新增素材。

### 5.6 查询策略

由于当前 `listDocuments` wrapper 不暴露复杂查询命令，第一版建议保持轻量：

- `listEntries`：读取 `coupleId: "main"` 后，在 repository 内过滤 `kind === "memory"` 或无 `kind`。
- `listSongs`：读取 `coupleId: "main"` 后，在 repository 内过滤 `kind === "song"`。
- `listTasks`：读取 `coupleId: "main"` 后，在 repository 内过滤 `kind === "task"`。

### 7.5 FeatureCard 视觉与情绪标准

`FeatureCard` 是首页情绪核心，不能做成普通 dashboard card。

必须包含：

- 标题：例如「点歌清单」
- 亲密副文案：例如「想听你唱」
- 数量或状态：例如「5 首想听」
- 中文小 mark：例如「歌」「约」「忆」
- tone：`primary` / `accent` / `neutral`

推荐结构：

```text
[歌] 点歌清单
     她想听的歌先放这里
     5 首想听
```

视觉要求：

- 像小便签 / 小入口，不像统计面板。
- 使用 `@include panel` 和 `@include pressable`。
- mark 使用 `--app-primary-soft` / `--app-accent-soft`。
- 数量文字使用 `--app-primary` / `--app-accent`。
- `FeatureCard` 的点击反馈使用 `--app-press-opacity` 与 `--app-press-scale`。
- 不出现“模块管理”“功能中心”“任务统计”等后台语气。

### 7.6 SongCard 情侣语义标准

`SongCard` 不能像普通歌单管理器。它表达的是“她想听我唱的歌”。

必须展示：

- 歌名
- 歌手或版本
- 想听理由
- 状态：想听 / 已唱 / 暂缓
- 心愿程度：很想听 / 想听 / 随时
- 操作：标记已唱 / 改回想听 / 编辑

推荐文案：

| 状态 | 卡片语气 |
|---|---|
| 想听 | `还想听`、`等你唱`、`很想听` |
| 已唱 | `已唱给她听`、`已收好` |
| 暂缓 | `暂缓一下`、`以后再唱` |

视觉要求：

- 状态标签像小便签，不像后台 badge。
- “标记已唱”是温柔动作，不是任务完成按钮。
- 已唱曲目保留在列表，不消失。
- 已唱后使用 `--app-success-soft` 做轻柔完成感。
- 暂缓使用 `--app-warning-soft`，不做警告式强提示。

### 7.7 TaskCard 情侣语义标准

`TaskCard` 不能像企业待办。它表达的是“两个人想一起做的小事”。

必须展示：

- 事项标题
- 备注
- 期待日期
- 完成状态
- 复选框
- 操作：勾选 / 取消完成 / 编辑

推荐文案：

| 状态 | 卡片语气 |
|---|---|
| 未完成 | `还没一起做`、`慢慢来` |
| 已完成 | `已经一起完成`、`已收好` |
| 无日期 | `不着急`、`随时都好` |

视觉要求：

- 复选框像小勾选贴纸，不像表单控件堆叠。
- 完成后轻轻淡化，不从列表消失。
- 完成态可以有轻柔划线或弱化，但必须可读。
- 标题不要使用过强删除线导致阅读困难。
- 期待日期像小纸条，不像 deadline。

### 8.10 页面级视觉补强标准

#### 首页

首页必须像“私密记忆停靠处”，不是工具入口页。

具体要求：

- Hero 文案保留温柔记录感。
- 三个入口卡片不要做成冷冰冰统计卡。
- 点歌入口偏红色线索。
- 必做事项入口偏蓝色线索。
- 回忆入口保留红蓝小人或“忆”字 mark。
- 数量展示是辅助，不是主视觉。
- 首页不使用“管理”“模块”“数据”等后台词。

推荐入口文案：

```text

她想听的歌先放这里
5 首想听

必做事项
我们想一起做的小事
8 件未完成

回忆记录

12 条记录
```

#### 点歌清单页

点歌清单要像“小小歌单”，不是音乐管理器。

具体要求：

- 顶部说明强调“想听你唱”。
- 筛选按钮像小纸条。
- 卡片状态标签像手写便签。
- 操作按钮文案保持亲密克制。
- 不出现“歌曲管理”“状态更新”“优先级”等后台词作为 UI 文案。

推荐空态：

```text
还没有想听的歌
可以先写一首她想听的，也可以写一首你想主动唱给她的。
按钮：加一首歌
```

#### 必做事项页

必做事项要像“我们的小约定”，不是企业待办。

具体要求：

- 顶部说明强调“不着急，慢慢完成”。
- 复选框视觉圆润，完成动作轻柔。
- 已完成事项保留在列表。
- 完成进度文案不要像 KPI。
- 不出现“任务管理”“效率”“截止日期”等工具化词。

推荐空态：

```text
还没有小约定
先写下一件想一起做的小事，慢慢来就好。
按钮：加一件事
```

#### 编辑页

编辑页要像“写一张小纸条”，不是后台表单。

具体要求：

- intro 卡片解释为什么写下它。
- label 简洁。
- placeholder 亲密但不油腻。
- 保存按钮明确。
- 删除区克制，不卖萌过度。

#### 删除确认

删除确认应明确、克制、保护数据：


```text
删除事项
这会删除这件计划事项。
```

回忆删除沿用现有逻辑。

## 11. 开发阶段计划

目标：

- 在 `love_entries` 中支持 `memory / song / task`。
- 旧回忆记录兼容。
- 点歌与事项不会混入回忆时间线。

任务：

1. 修改 `entries.ts`：
   - 新建回忆写入 `kind: "memory"`。
   - 读取旧数据时 `kind` 缺失按 `memory` 处理。
   - `listEntries()` 过滤掉 `kind === "song"` 和 `kind === "task"`。
   - 删除回忆仍清理云端文件。

   - `listSongs`
   - `getSong`
   - `createSong`
   - `updateSong`
   - `updateSongStatus`
   - `deleteSong`

   - `listTasks`
   - `getTask`
   - `createTask`
   - `updateTask`

   - `deleteTask`
4. 所有新增文档写入：

## 13. 风险与处理

### 13.1 同一 collection 多类型记录导致列表混入

风险：

- 如果 `listEntries` 不过滤 `kind`，点歌和事项会显示在回忆时间线。

处理：

- `entries.ts` 必须兼容过滤。
- 旧记录无 `kind` 视作回忆。
- 新建回忆写入 `kind: "memory"`。

---

### 13.2 客户端过滤在未来数据量大时不够稳

风险：

- `listDocuments` 先 `limit` 后客户端过滤，未来数据多时可能导致回忆列表数量不准确。

处理：

- Stage 1 可接受。
- 后续如数据增长，在 `cloudbase.ts` 增加受控 command helper 或专用 query 方法。
- 不在页面直接调用 `wx.cloud`。

---

### 13.3 Wot checkbox API 适配风险

风险：

- Wot UI checkbox 在 UniApp mp-weixin 中的 model/event 写法需要以当前版本为准。

处理：

- 优先用 Wot checkbox。
- 若验证失败，改为 tokenized 自定义复选框组件。
- 自定义组件仍必须通过 `scan:design-tokens`。

---

### 13.4 文案扫描误判

风险：

- `wanted / sung / paused` 等枚举可能被 UI 文案扫描误判，或错误显示到界面。

处理：

- 枚举只在代码和数据库中使用。
- UI 展示必须通过中文 label map。
- 页面上不直接输出 enum 原值。

---

### 13.5 设计稿 raw 色值误复制

风险：

- 设计稿中的暖纸、红、蓝、浅色卡片等如果直接复制为 hex，会违反设计系统规则。

处理：

- 开发时只使用本文件的 token 替换矩阵。
- `pnpm scan:design-tokens` 作为硬门禁。

---
